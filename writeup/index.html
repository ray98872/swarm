Engineered · case study

Research Agent Swarm

A multi-agent research system that answers a technical comparison question by fanning out to five specialist agents in parallel, then merging their findings into one cited report — all on free tiers, at £0 per query, with custom orchestration and no agent framework.

Try the live demo → Source on GitHub ← Portfolio

01 — Overview

What it does

You type something like “Should I migrate from Redis to Valkey?” An orchestrator dispatches five specialist sub-agents simultaneously. Each one retrieves information from a different kind of source, extracts a handful of concrete findings, and returns a confidence score. A sixth synthesis agent then merges everything into a structured, cited report: a verdict, a summary, pros, cons, risks and numbered citations.

The frontend doesn’t just wait for the answer. It opens a Server-Sent Events stream and shows each agent’s status live — grey while waiting, amber while searching, green when done with a confidence badge — so you watch the swarm think in real time before the report slides in.

02 — Architecture

Star fan-out, not a pipeline

orchestrator dispatch results → synthesis

The topology is a star: one orchestrator at the centre, five agents on the spokes, all launched at once with a single asyncio.gather. Results converge on one synthesis step. I chose this over a sequential pipeline for two reasons.

First, latency. The five agents are independent — the docs agent doesn’t need the benchmark agent’s output — so running them in series would just add their wall-clock times together. In a star, total time is roughly the slowest agent plus synthesis, not the sum.

Second, failure isolation. In a pipeline, a stalled stage blocks everything behind it. In a star, each agent has its own 15-second timeout; if one times out it returns an empty result with confidence: 0.0 and the synthesis step simply proceeds with the other four. Graceful degradation falls out of the topology for free.

I deliberately wrote the orchestration by hand rather than reaching for LangGraph or CrewAI. For a fan-out this simple, a framework would add concepts and indirection without removing real work — and the point of the project is to show I understand the control flow, not that I can configure someone else’s.

03 — Agent design

One contract, five specialists

Every agent returns the same shape. That uniform contract is what lets the orchestrator treat them interchangeably and lets the synthesis step reason about them side by side: 

@dataclass
class AgentResult:
    agent_name: str
    findings: list[str]
    citations: list[str]
    confidence: float   # 0.0–1.0
    elapsed_ms: int

A BaseAgent wraps each implementation’s real work in a timeout and a timer, so a subclass only writes retrieval + extraction and is guaranteed to return a valid AgentResult — even on failure. Each agent exists because it queries a different kind of source:

web_search

Broad, current coverage via the duckduckgo-search library — no API key, no quota. The generalist that catches whatever the specialists miss.

docs

Biases its query toward official documentation domains, then scrapes and de-tags the top pages with httpx + BeautifulSoup. The most authoritative source.

benchmark

Hunts for performance articles and extracts the numbers — throughput, latency, memory, measured deltas — rather than vibes.

community

Queries the Hacker News Algolia API (free, no auth) plus a discussion-biased web pass for lived developer experience: gotchas, migration pain, real consensus.

risk

Searches explicitly for downsides, breaking changes and caveats. It runs in parallel like the others; cross-agent contradiction resolution happens in synthesis, which is the one place that sees every output at once.

Each retrieval agent hands its gathered text to the fast model with a role-specific instruction and gets back a small JSON object of findings plus a self-assessed confidence. The extraction prompt is shared, so all five behave consistently and none can invent facts that aren’t in their sources.

04 — Model split

8B for retrieval, 70B for the merge

All six LLM calls go to Groq’s free tier, but not to the same model. The five sub-agents use llama-3.1-8b-instant; the synthesis agent uses llama-3.3-70b-versatile.

The reasoning is a cost/quality trade-off mapped onto the work each step actually does. Extraction is easy — pull concrete sentences out of supplied text — and it happens five times, in parallel, on the critical latency path. The 8B model is fast enough that it doesn’t become the bottleneck, and accurate enough for “copy the facts that matter.” Synthesis is hard — weigh conflicting evidence, resolve contradictions, attach citations, commit to a verdict — and it happens once, off the parallel path. That single call is where the larger model earns its extra latency.

Spending the big model only where judgement is required keeps the whole query inside Groq’s free 30-requests-per-minute limit (six calls per query) while still producing a report that reads like it was written by one careful analyst rather than five disconnected ones.

05 — Streaming

SSE, not WebSockets

The flow is two steps: the client POSTs the question to /query and gets a session_id, then opens an EventSource on /stream/{id}. The orchestrator is written as an async generator that yields events; the FastAPI layer turns each one into an SSE frame: 

event: agent_start    { "agent": "web_search" }
event: agent_done     { "agent": "web_search", "confidence": 0.82, "elapsed_ms": 1240 }
event: synthesis_start
event: report_ready   { ...full structured report... }

I used Server-Sent Events rather than WebSockets because the communication is one-directional: the server streams progress, the client only listens. SSE is plain HTTP — it works through the Fly.io proxy and GitHub Pages CORS without a second protocol, auto-reconnects in the browser, and needs no extra library. WebSockets would buy bidirectional messaging the app never uses, at the cost of more moving parts.

The one hard rule: the stream must never hang. Agents emit their events as they finish (fast agents report first), and the generator always ends with either report_ready or error — even if every agent fails — so the client is never left waiting on a connection that will never speak again.

06 — Performance

Parallel vs sequential

Because the orchestrator records each agent’s elapsed_ms, it can report the real parallel wall-clock time against a naive sequential estimate (the sum of all agent times). Representative numbers from local runs on the free tiers:

StageSequential (sum)Parallel (star)
5 retrieval agents~9.5 s~2.8 s
Synthesis (70B)~2.5 s~2.5 s
Total~12.0 s~5.3 s

Figures vary with query and network conditions; the demo shows the measured numbers for your actual run. The point is the shape: fan-out collapses five retrieval latencies into roughly one.

Cost per query: £0. Groq’s free tier covers the LLM calls, DuckDuckGo and the Hacker News Algolia API need no keys, the backend runs on a free always-on Fly.io VM, and the frontend is static on GitHub Pages. There is no paid tier or trial anywhere in the stack.

07 — What I’d do at scale

Where the demo’s shortcuts would break

The current build keeps sessions in an in-process dictionary and runs a single Fly.io worker. That’s the right call for a portfolio demo, but it’s also exactly where I’d change things if real traffic showed up:

  • Worker pool + Redis pub/sub. In-memory sessions tie the SSE stream to the one process that created it. Moving session state and agent events onto Redis pub/sub lets any worker serve any client’s stream, so I could scale horizontally behind the Fly.io proxy.
  • Persistent sessions & result caching. Store completed reports keyed by a normalised question, so identical or similar queries return instantly and cheaply instead of re-running the swarm.
  • Bounded concurrency & backpressure. The free Groq limit is fine for one user; under load I’d add a per-model rate limiter and a queue so bursts degrade predictably rather than tripping 429s mid-stream.
  • Retrieval hardening. Rotate search backends and cache fetched pages so a single provider’s rate limit doesn’t quietly drop an agent to zero confidence.
